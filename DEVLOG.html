Header Tasarımı — SVG Arch

Tasarım Kararı

Standart dikdörtgen header yerine, ortada aşağı doğru kavisli (arch/wave) bir header tasarlandı. Logo, bu kavisin tam ortasına yerleştirildi. Tasarım SVG ile çizildi — hiçbir CSS gradient veya clip-path kullanılmadı. 

/*
  Header şekli (temsili):
  ┌──────────── TAM GENİŞLİK ────────────┐  ← navHeight (85px)
  │  [ÜRÜNLERİMİZ]  [LOGO]  [HAKKIMIZDA] │
  └──────────╮               ╭───────────┘
              ╲      🍯      ╱
               ╰─── arch ───╯  ← waveDepth (90px) kadar iner
*/

// SVG path hesaplaması
const VB_W = 1440;
const VB_H = headerTheme.navHeight + headerTheme.waveDepth; // 85+90=175

// Beyaz dolgu path'i
const headerPath = [
  `M 0 0`,
  `L 1440 0`,
  `L 1440 85`,
  `L 900 85`,
  `C 820 85, 780 175, 720 175`,  // sağdan merkeze bezier
  `C 660 175, 620 85, 540 85`,   // merkezden sola bezier
  `L 0 85`,
  `Z`,
].join(" ");

Navigasyon Yapısı

  • Sol nav: yalnızca ilk link (ÜRÜNLERİMİZ)
  • Orta: logo için boşluk (logoWidth + 48px)
  • Sağ nav: kalan linkler + User / Search / Cart ikonları
  • Mobil: hamburger menü, arama ve sepet ikonları

Shop Layout & Header–İçerik Örtüşmesi

Hedef

Header'ın arch/wave kısmı, hero slider ve bölüm görsellerinin üstüne binmeli. Ama aynı zamanda, sayfa içeriği headerın arkasına gizlenmemeli.

Çözüm: fixed + paddingTop Stratejisi

  • Header: position: fixed → akıştan çıkar, her zaman üstte kalır
  • <main>: paddingTop = solidHeight (125px) → içerik "solid" bölümün altından başlar
  • Wave (90px): main'in padding'inin üzerinden içeriğe biner — efekt sağlanır
// app/(shop)/layout.tsx
import { headerTheme } from "@/lib/theme";
const MAIN_PT = headerTheme.solidHeight; // 125 px

export default function ShopLayout({ children }) {
  return (
    <>
      <Header />
      <main style={{ paddingTop: MAIN_PT }}>{children}</main>
      <Footer />
    </>
  );
}
⚠️ BAKIM NOTU: solidHeight = announcementHeight + navHeight. Bu değerlerden biri değiştiğinde solidHeight de manuel güncellenmeli!

Hero Slider — Swiper.js ile İmaj & Gradient Modu

Problem: Geçişler Kötü Görünüyordu

Sadece görsel tanımlandığında geçişlerde beyaz flaş oluşuyor, üzerinde istenmeyen overlay/fade görünüyordu.

Çözüm: EffectFade + crossFade + Dual Branch

  • effect="fade" + fadeEffect={{ crossFade: true }} → geçişlerde arka planın görünmesini önler
  • Image branch: slide.image varsa → sadece tam ekran görsel, hiç overlay yok
  • Gradient branch: slide.image yoksa → gradient arka plan + metin + istatistikler
function SlideContent({ slide }) {

  // ── IMAGE ONLY BRANCH ──────────────────────────────────
  if (slide.image) {
    return (
      <section className="hero-slide-section relative overflow-hidden bg-black">
        <img
          src={slide.image}
          className="absolute inset-0 w-full h-full object-cover"
        />
        </* Alt dekoratif dalga SVG */>
      </section>
    );
  }

  // ── GRADIENT + TEXT BRANCH ─────────────────────────────
  return <section style={{ background: `linear-gradient(...)` }}>...</section>;
}

Yükseklik Sorunu: Swiper Fade'de Container Collapse

Fade modunda slide'lar position: absolute olduğu için Swiper container'ı JS çalışana kadar height: 0'a düşüyor; beyaz alan görünüyordu.

✓ ÇÖZÜM: Hem wrapper div'e hem de <Swiper>'a style={{ minHeight: sliderTheme.height }} eklendi. Responsive yükseklik için CSS @media kuralları da eklendi. 
// Responsive yükseklik — theme değerlerinden
<style>{`
  .hero-slide-section { min-height: ${sliderTheme.heightMobile}px; }
  @media (min-width: 768px) {
    .hero-slide-section { min-height: ${sliderTheme.height}px; }
  }
`}</style>

// sliderTheme.slides içeriği theme.ts'ten gelir
const slides = sliderTheme.slides as unknown as Slide[];

as const ile tanımlanan slides array'indeki literal union tipler gradientFrom? gibi optional alanları bulamıyordu.

// lib/dia-erp/client.ts — JWT token yönetimi
private async getToken(): Promise<string> {
  const now = Date.now();
  // Token geçerliliği 5 dakikadan fazlaysa önbellekten dön
  if (tokenCache && tokenCache.expiresAt - now > 5 * 60 * 1000) return tokenCache.token;
  // Yenile: POST /api/auth/token { api_key }
  const res = await fetch(`${this.baseUrl}/api/auth/token`, {
    method: "POST",
    body: JSON.stringify({ api_key: this.apiKey }),
  });
  const { token, expires_in } = await res.json();
  tokenCache = { token, expiresAt: now + parseExpiresIn(expires_in) };
  return token;
}

QNBPay Token Servisi — Server-Side JWT

Neden Server-Side?

QNBPay token almak için app_secret gerekir. Bu değerin tarayıcıya gitmesi PCI-DSS'i ihlal eder. Token endpoint'i sunucuda çağrılır, dönen JWT yalnızca Authorization: Bearer header'ı olarak kullanılır.

Keşfedilen Sorun: Yanıt Yapısı İç İçe

QNBPay'in token yanıtı response.data.token değil, response.data.data.token şeklinde iç içe geliyor.

✓ ÇÖZÜM: response.data?.data?.token ?? response.data?.token — her iki yapıyı da destekler. 
// lib/payment/qnb-pay.ts
private async getToken(): Promise<string> {
  const response = await axios.post(
    `${this.baseUrl}/ccpayment/api/token`,
    { app_id: this.appId, app_secret: this.appSecret },
    { headers: { "Content-Type": "application/json" }, timeout: 10000 },
  );

  // Yanıt: { status_code: 100, data: { token: "eyJ..." } }
  const token = response.data?.data?.token ?? response.data?.token;
  if (response.data.status_code !== 100 || !token) {
    throw new Error(`Token alınamadı: ${response.data.error_message}`);
  }
  return token;
}

AES-256-CBC Hash Key Formülü

Neden Gerekli?

QNBPay, form verilerinin değiştirilmediğini doğrulamak için şifrelenmiş bir hash_key ister. Bu key sunucuda hesaplanır — app_secret tarayıcıya gitmez.

Formül (QNBPay JS örneğinden Node.js'e uyarlandı)

  1. 16 karakterlik rastgele iv (SHA1 → hex)
  2. 4 karakterlik rastgele salt (SHA1 → hex)
  3. password = SHA1(app_secret) → hex
  4. key = SHA256(password + salt) → hex, ilk 32 karakter
  5. AES-256-CBC şifreleme: plaintext = total|installments|currency|merchant_key|invoice_id
  6. Sonuç: iv:salt:encrypted, tüm / → __
private generateHashKey(data: string): string {
  const iv = crypto.createHash("sha1")
    .update(Math.random().toString()).digest("hex").substring(0, 16);

  const salt = crypto.createHash("sha1")
    .update(Math.random().toString()).digest("hex").substring(0, 4);

  const password = crypto.createHash("sha1")
    .update(this.appSecret).digest("hex");

  const key = crypto.createHash("sha256")
    .update(password + salt).digest("hex").substring(0, 32);

  const cipher = crypto.createCipheriv(
    "aes-256-cbc",
    Buffer.from(key, "utf8"),
    Buffer.from(iv, "utf8"),
  );

  const encrypted =
    cipher.update(data, "utf8", "base64") + cipher.final("base64");

  return `${iv}:${salt}:${encrypted}`.replaceAll("/", "__");
}
⚠️ ÖNEMLI: Hash'e dahil edilen veri: total|1|TRY|merchant_key|orderNumber. Kart verisi hash'e dahil değildir — hash yalnızca sipariş bilgilerini doğrular.

// 🔒 Kart verisi sunucuya hiç gitmez — doğrudan QNBPay'e form POST edilir
function submitToQNBPay(payload: Smart3DFormPayload, card: CardData) {
  const form = document.createElement("form");
  form.method = "POST";
  form.action = payload.endpoint; // https://portal.qnbpay.com.tr/ccpayment/api/paySmart3D

  const fields = {
    ...payload,  // merchant_key, authorization, hash_key, items …
    // Kart alanları — yalnızca browser'da, sunucuya uğramaz
    cc_holder_name: card.cc_holder_name,
    cc_no:          card.cc_no.replace(/\s/g, ""),
    expiry_month:   card.expiry_month,
    expiry_year:    card.expiry_year,
    cvv:            card.cvv,
  };

  for (const [name, value] of Object.entries(fields)) {
    if (!value) continue;
    const input = document.createElement("input");
    input.type = "hidden";
    input.name = name;
    input.value = value;
    form.appendChild(input);
  }

  document.body.appendChild(form);
  form.submit(); // ← browser direkt QNBPay'e POST atar
}
